Specifications

Write concise markdown specs. Expand them with AI.
Build the application in one shot.

Workflow

1
Create
bin/create_spec.sh MyProject
Scaffolds a spec directory with all template files
2
Edit
Edit files in MyProject/
Fill in INTENT, ARCHITECTURE, add SCREEN and FEATURE files
3
Validate
bin/validate.sh MyProject
Checks required files, naming, fields. Exit 0 = ready
4
Convert
bin/convert.sh MyProject
Generates expansion prompt for AI agent
5
Build
bin/build.sh MyProject
Tags commit, generates complete build prompt
6
Promote
Copy to own repo
Spec directory becomes the project's documentation

After a build, iterate: edit specs → validate → build. Each build creates an annotated git tag that permanently records the spec state.

File Reference

Required
METADATA.md
Project identity: name, status, description
README.md
One-line project description
INTENT.md
Why this project exists, scope boundaries
ARCHITECTURE.md
Modules, routes, directory layout
Conditional
DATABASE.md
Tables and columns (if project has a DB)
UI.md
Shared UI patterns (if project has a UI)
SCREEN-{Name}.md
Per-screen: route, layout, interactions
FEATURE-{Name}.md
Cross-cutting behavior: trigger, sequence

Conventions

Open Questions
All spec files except README, METADATA, INTENT end with ## Open Questions
Naming
Uppercase with hyphens: SCREEN-Dashboard.md, FEATURE-Scan.md
Concise Specs
Write tables and bullets. CONVERT.md rules expand them during conversion.
Stack Patterns
GLOBAL_RULES/stack/ has technology patterns — don’t repeat in specs.
