GEM Generic Enterprise Management

One platform.
Every project.

GEM enforces a standard contract across every project you run. In return, the platform manages operations, scheduling, health, logging, and AI context automatically — with no per-project configuration.

Control Panel

Project Dashboard

Real-time view of every project — service state, git status, compliance level, last run, health. One screen for the entire portfolio.

One-click Operations

Every script registered with # CommandCenter Operation appears as a button. Start, stop, build, deploy — same UI for every stack.

Run History & Logs

Execution history with exit codes, timestamps, and stdout/stderr. Logs are persisted to logs/ and accessible from the dashboard.

Inline Config Editing

Edit project metadata fields directly in the UI. Changes sync bidirectionally between the database and METADATA.md on disk.

Service Catalog

Auto-Discovery

Drop a METADATA.md in a directory. The platform discovers the project, reads its port, stack, status, and health endpoint — no registration step.

Health Monitoring

Continuous HTTP polling of every service's /health endpoint. Response time tracked. State transitions (UP/DOWN/MIXED) surfaced immediately.

Uptime Tracking

Rolling uptime percentage per service. Alerts with snooze. You see the failure before a user does.

Endpoint Registry

Service endpoints declared in AGENTS.md are extracted and indexed. Services discover each other through the catalog — no hardcoded URLs.

Scheduler

Cron-based Scheduling

Add a # Schedule: 0 2 * * * header to any script. The platform registers it, fires it on time, and tracks last/next run — no system cron needed.

Missed-run Recovery

On startup, the scheduler catches up any jobs that fired while the platform was down. No silent missed runs.

Standard Maintenance Hooks

bin/daily.sh and bin/weekly.sh are the standard slots for recurring work. Platform discovers and runs them — they appear in the schedule view.

Integrated Logging

Zero-config Log Files

Source bin/common.sh (or import common.py) and every script gets a timestamped log file in logs/ automatically.

Structured Start/Stop Markers

Every script logs [ProjectName] Starting: script.sh and a completion marker with exit status. No log archaeology required.

Signal Handling

SIGTERM trap is wired automatically. Graceful shutdown, cleanup, and final log entry happen without any script-level code.

Env & Secrets Loading

common.sh loads .env and .secrets before execution. Scripts get their config — no boilerplate, no drift between scripts.

Event System

Operation Events

Every script execution emits started, completed, and failed events — available on the monitoring timeline in real time.

Service State Events

Health transitions (UP↔DOWN) fire events. Downstream consumers — alerts, dashboards, webhooks — react without polling.

Git Events

Push and commit events from the dashboard are logged to the event stream and linked to tickets and transaction log entries.

Scheduler Events

Fired and missed-run events per job. Visible in the platform event timeline alongside service and operation events.

Compliance & Governance

Conformity Levels

IDEA → PROTOTYPE → ACTIVE → PRODUCTION. Each level has enforced requirements. bin/validate.sh checks them — same check locally and in CI.

Rules Propagation

Update a rule or template once in RulesEngine/. bin/update_projects.sh pushes the change to every registered project.

Auto-detection Flags

Platform detects has_git, has_venv, has_tests, has_docs, has_specs from file presence. Features enable themselves.

Project Health Score

Composite score from compliance level, git activity, service health, test presence, docs, and env drift. One number per project.

Development Workflow

Feature Branch Pipeline

bin/build.sh clones the repo, creates the feature branch, and generates the build prompt. bin/merge.sh squash-merges it back. Same flow for every project.

Kanban Tickets

Per-project ticket board with configurable states (IDEA → IN DEVELOPMENT → DONE). State transitions have validation rules. Tickets link to git commits and spec changes.

Transaction Log

Immutable log linking spec commits, code changes, and ticket transitions. Full traceability from idea to deployed change.

Spec-to-Code Traceability

Spec directories in Specifications/ are linked to their code projects. The platform knows which spec version drove which build.

AI Agent Context

AGENTS.md Contract

Every project carries AGENTS.md: dev commands, endpoints, architecture, and the full CLAUDE_RULES block. Any AI agent reads it and starts working.

Injected Rules Block

Platform-wide behavioral rules are injected into AGENTS.md as a versioned block. Rules update automatically when bin/ProjectUpdate.sh runs.

Bookmark Extraction

Links declared under ## Bookmarks in AGENTS.md are surfaced in the dashboard alongside the project — no duplication needed.

Publishing

Portfolio Site Generation

Projects with show_on_homepage = true are rendered into a static portfolio site. Card content pulled from METADATA.md fields.

Documentation Generation

bin/build_documentation.sh produces doc/index.html from project sources. index.html at root redirects to it — standard entry point for every project.

GitHub Pages Publishing

One command publishes the portfolio to GitHub Pages. No separate CI pipeline required.
