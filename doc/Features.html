The GEM Rules Framework

A standard for building software projects that are observable, operable, and self-describing — so that any tool, team member, or AI agent can find, run, and manage them without custom configuration.

Origin: The GEM (Generic Enterprise Management) Rules Framework was first defined circa 2012 to automate the management of large infrastructure estates — standardizing how services expose their operations, health, and identity so that a central platform could monitor and control them at scale. These same principles now govern every project built under this system.
The Framework

What a Conformed Project Provides

Every project that follows GEM Rules exposes a standard set of endpoints and behaviours. You don't configure the platform to understand your project — the project describes itself.

CapabilityWhat It DoesWhy It Matters
/health Returns a live status check — is the service running and healthy? The dashboard knows instantly if something is wrong, without polling logs
METADATA.md Machine-readable identity: name, port, status, stack, description, git repository The platform discovers and registers the project automatically — no manual entry
bin/start.sh A single command to start the service, regardless of technology Any operator — human or automated — can start the project the same way
bin/stop.sh Graceful shutdown with signal handling Clean stops prevent data corruption and port conflicts
bin/test.sh Runs the project's test suite in a single command Continuous integration and manual validation work identically
doc/ Auto-generated documentation available at a known URL Documentation is always current — it is produced from the code, not written separately
AGENTS.md Context file that tells an AI agent what the project does and how to work in it Any AI agent can pick up and continue work on the project without briefing
The key insight: when every project uses the same layout, naming, and endpoints, a single dashboard can manage all of them. You stop maintaining integration glue and start managing outcomes.
For Software Projects

What This Gives You

Built for teams who want consistency without bureaucracy. These features apply immediately when a project adopts GEM Rules — no extra tooling required.

Observability

Dashboard Discovery

The GAME dashboard scans your project folder and registers every conformed project automatically. Name, description, status, and port appear in the UI without any configuration step.

Observability

Live Health Monitoring

Every project exposes a /health endpoint. The dashboard polls it and shows a green or red status at a glance — you see what is running before you try to use it.

Operations

One-Command Start and Stop

No matter what language or framework the project uses, bin/start.sh and bin/stop.sh work the same way. New team members and automated tools never have to read a README to run something.

Operations

Centralised Log Access

Logs are written to a standard logs/ directory with consistent naming. Any monitoring tool can find and tail them without project-specific configuration.

Development

AI-Ready Context

Every project contains an AGENTS.md file that gives an AI assistant full context about what the project does, how to run it, and what endpoints it exposes. Hand off to an AI agent at any point without a lengthy briefing.

Development

Feature Branch Builds

New features are developed on an isolated branch, named and tracked by the platform. You get a fully running version to test before anything reaches your main codebase. Merge is a single action — the platform handles the git operations invisibly.

Quality

Conformance Validation

A built-in validator checks whether a project meets the platform standard at each level — IDEA, PROTOTYPE, ACTIVE, PRODUCTION. You see exactly what is missing and what to fix, in plain language.

Quality

Consistent Testing Entry Point

bin/test.sh is required in every project. CI pipelines, the dashboard, and manual validation all call the same script. A passing test on your machine means the same thing everywhere.

Governance

Spec-to-Build Traceability

Every build is tagged in the specification repository with a timestamp and commit reference. You can diff any two builds, restore a previous spec, or audit exactly what changed between versions — without digging through commit history manually.

Governance

Rules Propagation

When platform standards change — security rules, shared utilities, documentation templates — a single command pushes the update to every conformed project. No project falls behind because someone forgot to apply a change.

Portability

Self-Contained Documentation

Each project generates its own browsable documentation in doc/. It works from the file system, a local server, or a hosted URL — no build server required. Documentation travels with the code.

Portability

Stack-Agnostic Standard

GEM Rules work across Python, Node, Go, shell scripts, or any other stack. The standard describes behaviour, not implementation. A Python service and a shell automation tool look identical to the platform.

Beyond Software

Procedural and Non-Software Projects

GEM Rules were designed for infrastructure automation — not just web applications. Any repeatable procedure can be a conformed project.

Automation

Scripted Workflows as First-Class Projects

A data pipeline, a nightly report generator, or an infrastructure maintenance script can all be conformed projects. They get a dashboard entry, a health check, a start command, and version control — the same as any web service.

Automation

Uniform Operational Interface

Whether a "project" sends emails, processes files, or provisions cloud resources, the operator interface is always the same: start, stop, test, check health. No bespoke runbooks for each process.

Scale

Platform-Level Visibility Across All Projects

The dashboard lists everything — software, scripts, and procedures — in one place. Status, port, description, and last-known state are visible without opening each project. Managing twenty projects takes the same effort as managing two.

Scale

AI-Assisted Maintenance at Scale

Because every project is self-describing, an AI agent can be pointed at any of them and immediately understand what it does, how to run it, and how to modify it. Maintenance work that would require a specialist for each system can be handled by a single generalised agent.

In short: GEM Rules turn a collection of individual projects into a managed platform. Each project contributes its identity, operations, and health to a shared system — and the platform gives back visibility, automation, and AI-assisted development that would otherwise require custom integration for each project.
