# Documentation Build Standard

**Version:** 20260321 V1
**Scope:** All projects — defines how documentation is built, structured, and published.

---

## Output Standard

Every project must produce `doc/index.html` as its primary documentation artifact.

| Pattern | Use case |
|---------|----------|
| `doc/index.html` | Standard — all new projects |
| `doc*/index.htm*` | Legacy projects with non-standard output paths |
| `doc/index_<name>.html` | Additional artifacts for the same project (specify in project spec) |

The `doc/` tree must be **fully portable** — copyable to any server (GAME local, GitHub Pages,
or any static host) without modification. All paths must be relative and self-contained within
`doc/`. No links may escape `doc/` using `../` traversal.

---

## Source Files

Documentation source files that live in `doc/` must be prefixed `DOC-` to prevent name
collisions with same-named files in the project root or spec directories.

| Convention | Example |
|------------|---------|
| `DOC-<NAME>.md` | `DOC-ENGINEERING-RULES.md`, `DOC-PROMOTE.md` |

The build script strips the `DOC-` prefix when deriving the internal guide key.

---

## Build Script

Every project must have `bin/build_documentation.sh`:

```
#!/bin/bash
# CommandCenter Operation
# Name: Build Doc
# Category: Operations
```

The script must:
1. Accept `--theme=<name>` (default: `slate`)
2. Concatenate `doc/styles/themes/<theme>.css` + `doc/styles/spec-base.css` → `doc/styles/spec.css`
3. Build `doc/index.html` (and any additional artifacts)
4. Write all output inside `doc/` — never outside it

GAME calls `bin/build_documentation.sh` via CommandCenter to rebuild docs on demand.
Multiple artifacts per project are built by the same script in sequence.

## Two Documentation Patterns

### Process-based (no web server)

Reference implementation: `Specifications/bin/build_documentation.py`

Used for projects that document a workflow or tool pipeline rather than a running web app.
Characteristics:
- Single `doc/index.html` file — all content rendered in-browser from inlined JS data
- Sidebar navigation with step sub-items and script man-pages
- Workflow diagrams (`wf-diagram`, `wf-row`, `wf-box`, `wf-arr` CSS classes)
- Guide content rendered via `marked.js` from inlined markdown strings
- No server dependency — opens directly in a browser from the filesystem

To build a new process-based doc, copy `Specifications/bin/build_documentation.py` as the
starting point. The layout, CSS classes, and JS patterns are the canonical reference.

### Server-based (web application)

Used for projects with a running Flask/web server. The documentation may link to live
endpoints or be generated by the server itself. Structure is otherwise the same — `doc/`
tree, `spec.css`, portable.

## Portability Requirements

Because `doc/` must be copyable to GitHub Pages or the GAME local server:

1. **No `../` escaping** — project viewer iframes must be copied into `doc/projects/<name>/`
   during the build, not linked from sibling directories.
2. **No absolute filesystem paths** — all `href` and `src` attributes use relative paths.
3. **CDN resources are acceptable** — `https://cdn.jsdelivr.net/...` works on all servers.
4. **CSS is generated at build time** — `doc/styles/spec.css` is always built fresh; copy it
   with `doc/` when deploying.

---

## Copying to a Server

```bash
# Copy doc/ to GitHub Pages branch root
cp -r doc/* /path/to/gh-pages/

# Copy to GAME local server static directory
cp -r doc/* ~/projects/GAME/static/docs/Specifications/
```

Everything needed is inside `doc/` — styles, images, projects/, index.html.

---

## Theming

See `RulesEngine/DOCUMENTATION_BRANDING.md` for the full color/typography standard.

Default theme for new projects: **slate** (charcoal sidebar, teal accent, warm off-white content).

Theme files live in `doc/styles/themes/`. The active theme is built into `doc/styles/spec.css`
at build time — never edit `spec.css` directly.
