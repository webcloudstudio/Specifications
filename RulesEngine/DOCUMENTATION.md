# Documentation Build Standard

**Version:** 20260402 V2
**Scope:** All projects — defines how documentation is built, structured, and published.

---

## Output Standard

Every project must produce `docs/index.html` as its primary documentation artifact.

| Pattern | Use case |
|---------|----------|
| `docs/index.html` | Standard — all new projects |
| `docs/Features.html` | Features grid (if applicable) |
| `docs/white-paper.html` | Methodology white paper (if applicable) |
| `docs*/index.htm*` | Legacy projects with non-standard output paths |
| `docs/index_<name>.html` | Additional artifacts for the same project (specify in project spec) |

The `docs/` tree must be **fully portable** — copyable to any server (GAME local, GitHub Pages,
or any static host) without modification. All paths must be relative and self-contained within
`docs/`. No links may escape `docs/` using `../` traversal.

---

## Source Files

Documentation source files that live in `docs/` must be prefixed `DOC-` to prevent name
collisions with same-named files in the project root or spec directories.

| Convention | Example |
|------------|---------|
| `DOC-<NAME>.md` | `DOC-ENGINEERING-RULES.md`, `DOC-PROMOTE.md` |

The build script strips the `DOC-` prefix when deriving the internal guide key.

---

## Build Script

Every project must have `bin/build_documentation.sh`:

```
#!/bin/bash
# CommandCenter Operation
# Name: Build Doc
# Category: Operations
```

The script must:
1. Accept `--theme=<name>` (default: `slate`)
2. Concatenate `docs/styles/themes/<theme>.css` + `docs/styles/spec-base.css` → `docs/styles/spec.css`
3. Build `docs/index.html` (and any additional artifacts)
4. Write all output inside `docs/` — never outside it

GAME calls `bin/build_documentation.sh` via CommandCenter to rebuild docs on demand.
Multiple artifacts per project are built by the same script in sequence.

## Two Documentation Patterns

### Process-based (no web server)

Reference implementation: `Specifications/bin/build_documentation.py`

Used for projects that document a workflow or tool pipeline rather than a running web app.
Characteristics:
- Single `docs/index.html` file — all content rendered in-browser from inlined JS data
- Sidebar navigation with step sub-items and script man-pages
- Workflow diagrams (`wf-diagram`, `wf-row`, `wf-box`, `wf-arr` CSS classes)
- Guide content rendered via `marked.js` from inlined markdown strings
- No server dependency — opens directly in a browser from the filesystem

To build a new process-based doc, copy `Specifications/bin/build_documentation.py` as the
starting point. The layout, CSS classes, and JS patterns are the canonical reference.

### Server-based (web application)

Used for projects with a running Flask/web server. The documentation may link to live
endpoints or be generated by the server itself. Structure is otherwise the same — `docs/`
tree, `spec.css`, portable.

## Multi-Page Documentation

When a project has multiple HTML pages (e.g., index.html, Features.html, white-paper.html),
all pages must:

1. **Link spec.css** — `<link rel="stylesheet" href="styles/spec.css">`
2. **Light body background** — `body { background: var(--c-bg); color: var(--c-text); }`
   The body is ALWAYS light. Never use `body.gem-topbar` on sub-pages — it turns the
   entire page dark and makes all content text invisible.
3. **Dark header bar confined to the header element** — the header gets its own dark background:
   ```html
   <body>
   <header class="page-header gem-header">
     <div class="gem-header-left">
       <span class="gem-header-logo">ProjectName</span>
       <span class="gem-header-title">Page Title</span>
     </div>
     <nav class="gem-header-nav">
       <a href="index.html">Documentation</a>
       <a href="Features.html">Features</a>
     </nav>
     <span class="gem-header-copyright">Author &middot; Org</span>
   </header>
   ```
   Add this CSS to confine the dark background:
   ```css
   .page-header { background: var(--c-topbar-bg); border-bottom: 1px solid var(--c-side-border); }
   ```
4. **Mark the active page** — inline `style="border-bottom-color: var(--c-accent);"` on the current nav link
5. **Use CSS variables for colors** — never hardcode color values; use `var(--c-accent)`, `var(--c-text)`, etc.

The sidebar (dynamic content, script discovery, project listing) lives only in `index.html`.
Sub-pages use the header bar for consistent chrome and inter-page navigation.

**Readability rule:** dark text on light backgrounds, white text on dark backgrounds — always.
Never use `--c-text`, `--c-h1`, or `--c-h3` on a dark surface. See `DOCUMENTATION_BRANDING.md`
for the full readability contract.

## Script Man Pages

When the main documentation page renders a script, it uses man-page format with standard sections:

| Section | Source header key | Content |
|---------|-------------------|---------|
| NAME | (auto) | `filename — description` |
| SYNOPSIS | `Usage:` or `Synopsis:` | Command invocation forms |
| DESCRIPTION | `_desc` prose + `Checks:` / `Workflow:` | What the script does |
| OPTIONS | `Arguments:` or `Options:` | Flags and parameters |
| MODES | `Modes:` | Operational modes |
| FILES | `Writes:` / `Creates:` / `Reads:` | Input/output files |
| EXAMPLES | `Examples:` | Usage examples |
| EXIT STATUS | `Exit codes:` | Return codes and meanings |

Script headers use `# Section Name:` format (title case, colon-terminated). The build
script maps these to uppercase man-page conventions automatically.

## Portability Requirements

Because `docs/` must be copyable to GitHub Pages or the GAME local server:

1. **No `../` escaping** — project viewer iframes must be copied into `docs/projects/<name>/`
   during the build, not linked from sibling directories.
2. **No absolute filesystem paths** — all `href` and `src` attributes use relative paths.
3. **CDN resources are acceptable** — `https://cdn.jsdelivr.net/...` works on all servers.
4. **CSS is generated at build time** — `docs/styles/spec.css` is always built fresh; copy it
   with `docs/` when deploying.

---

## Copying to a Server

```bash
# Copy docs/ to GitHub Pages branch root
cp -r docs/* /path/to/gh-pages/

# Copy to GAME local server static directory
cp -r docs/* ~/projects/GAME/static/docs/Specifications/
```

Everything needed is inside `docs/` — styles, images, projects/, index.html.

## Theming

See `RulesEngine/DOCUMENTATION_BRANDING.md` for the full color/typography standard.

Default theme for new projects: **slate** (charcoal sidebar, teal accent, warm off-white content).

Theme files live in `docs/styles/themes/`. The active theme is built into `docs/styles/spec.css`
at build time — never edit `spec.css` directly.
