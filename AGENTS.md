# AGENTS.md — Specifications Repository

THIS REPOSITORY IS EXEMPT FROM CLAUDE_RULES INJECTION — IT IS NOT A CODE REPOSITORY

This repository contains project specifications only.
The prototyper tooling lives in the **prototyper** repository at `../prototyper/`.

---

## Purpose

Each subdirectory is a project specification — a set of concise markdown files that define what to build. The prototyper's `bin/` scripts read specs from this directory via the `specification_directory: ../Specifications` field in `../prototyper/METADATA.md`.

When working in this directory you are **authoring specifications**, not building code. Your job is to write the WHAT precisely enough that the build pipeline can infer the HOW.

---

## Specification Authoring Rules

### 1. Write the WHAT, not the HOW

The author writes column names, screen layouts, trigger conditions, numbered sequences, and interactions. The build pipeline (via `oneshot_build_rules.md` + stack files) expands types, defaults, constraints, HTMX attributes, error handling, and test structure. **Do not duplicate what the stack files already define.**

- Write: `created_at | TEXT | timestamp`
- Not: `created_at | TEXT | DEFAULT datetime('now') NOT NULL`

- Write: `Button → POST /save → reload table`
- Not: `Button → hx-post="/save" hx-target="#table" hx-swap="outerHTML"`

### 2. Be concise by default

Use tables, bullet lists, and short imperative phrases. Complete sentences are not required. Prose belongs in `README.md` and `INTENT.md` only — specification files (SCREEN-*, FEATURE-*, DATABASE, ARCHITECTURE) should read like structured notes.

### 3. Every spec file ends with `## Open Questions`

All specification files except `README.md`, `METADATA.md`, and generated files (`IDEAS.md`, `REFERENCE_GAPS.md`, `SPEC_SCORECARD.md`) must end with:

```markdown
## Open Questions
- None.
```

During authoring, put all unresolved decisions here rather than leaving placeholder text in the body. A question answered is a question removed. An empty section reads `- None.`

### 4. Mark speculative features with `[ROADMAP]`

Features that are intended but not yet specified belong in the relevant section with a `[ROADMAP]` prefix. Do not write implementation detail for roadmap items — just preserve the intent and any open questions.

```markdown
| [ROADMAP] Email notifications | Send digest on daily close | |
```

The build pipeline will implement a minimal stub for roadmap items.

### 5. Use "specification" not "spec" in prose

"specification" is the correct word in documentation, comments, and instructions. "spec" is acceptable only in script names (e.g. `spec_iterate.sh`).

### 6. Do not create build-phase files during authoring

These files are generated by tooling **after** the specification is ready to build. Do not create them manually while actively authoring:

| File | Created by |
|------|-----------|
| `REFERENCE_GAPS.md` | `bin/spec_iterate.sh` |
| `SPEC_SCORECARD.md` | `bin/spec_iterate.sh` |
| `SPEC_ITERATION.md` | `bin/spec_iterate.sh` |
| `SCREEN-NNN-*.md` | Post-build iteration only |
| `FEATURE-NNN-*.md` | Post-build iteration only |
| `PATCH-NNN-*.md` | Post-build iteration only |
| `AC-NNN-*.md` | Post-build iteration only |

## Specification File Types

| File | Purpose | Required |
|------|---------|----------|
| `METADATA.md` | Project identity: `name`, `display_name`, `short_description`, `status`, `stack` | Yes |
| `README.md` | One-line description + `## Intent` section (why it exists, who it's for) | Yes |
| `INTENT.md` | Goals, constraints, success criteria — standalone intent document | Yes |
| `ARCHITECTURE.md` | Modules, responsibilities, directory layout | Yes |
| `FUNCTIONALITY.md` | High-level feature summary — what the app does | Yes |
| `DATABASE.md` | Table names, column names, types, brief notes (schema only) | If has DB |
| `UI-GENERAL.md` | Shared UI patterns used across screens | If has UI |
| `SCREEN-{Name}.md` | Per-screen: route, layout sections, interactions | If has UI |
| `FEATURE-{Name}.md` | Per-feature: trigger, numbered sequence, reads, writes | As needed |
| `IDEAS.md` | Feature backlog — written by `tran_logger.sh` or manually | No |
| `ACCEPTANCE_CRITERIA.md` | Acceptance criteria for the current build | No |

## Per-File Content Rules

### METADATA.md
Key-value format (not YAML). Required fields:
```
name: ProjectName
display_name: Human Readable Name
short_description: One sentence.
status: IDEA|PROTOTYPE|ACTIVE|PRODUCTION|ARCHIVED
stack: Python/Flask/SQLite
```
Optional: `port`, `git_repo`, `health`, `version`, `updated`.

### README.md
```markdown
One-line description of the project.

## Intent
Why this project exists and who it is for.
```
No `## Open Questions` section.

### ARCHITECTURE.md
Module names, their single responsibility, and the directory layout. Route table if the project has HTTP routes. No implementation detail — name the modules, not the function signatures.

### DATABASE.md
Tables and columns only. Author writes: table name, column names, types, and a brief note. Do not write defaults, constraints, or indexes — those are inferred by the build pipeline from the stack rules.

```markdown
## tablename
| Column | Type | Notes |
|--------|------|-------|
| id | INTEGER PK | |
| name | TEXT | unique |
| status | TEXT | IDEA/ACTIVE/DONE |
| created_at | TEXT | timestamp |
```

### SCREEN-{Name}.md
```markdown
## Route
GET /path

## Layout
- Section one: description
- Section two: description

## Interactions
| Action | Trigger | Result |
|--------|---------|--------|
| Save | Button click | POST /save, reload section |

## Open Questions
- None.
```

### FEATURE-{Name}.md
```markdown
## Trigger
What initiates this feature (user action, schedule, event, etc.)

## Sequence
1. Step one
2. Step two
3. Step three

## Reads
- table.column

## Writes
- table.column ← value

## Open Questions
- None.
```

### UI-GENERAL.md
Component name, purpose, and key visual details. Do not write Bootstrap class names or HTMX attributes — those are expanded by the build pipeline.

## Lifecycle: When to Signal Readiness

1. **Authoring** — edit spec files, add Open Questions, mark roadmap items
2. **Review** — run `bash ../prototyper/bin/validate.sh <ProjectName>` to check for errors
3. **Ready to build** — tell the operator the specification is complete; they run `oneshot.sh`
4. **Post-build iteration** — numbered ticket files (`SCREEN-NNN-*`, etc.) are created here; `iterate.sh` applies them

Do not run `spec_iterate.sh` or `iterate.sh` until the operator signals the specification is ready to build or iterate against an existing codebase.

## Running Tooling (from the prototyper)

All tooling runs from `../prototyper/`. The scripts find this directory automatically via `specification_directory:` in prototyper's `METADATA.md`.

```bash
cd ../prototyper

# Scaffold a new specification directory here
bash bin/setup.sh <ProjectName>

# Validate a specification
bash bin/validate.sh <ProjectName>

# Generate build prompt (run when spec is ready)
bash bin/oneshot.sh <ProjectName> > ../Specifications/<ProjectName>/oneshot-prompt.md

# Gap analysis and quality scoring (post-authoring)
bash bin/spec_iterate.sh <ProjectName>
```
