Workflow 1

Prototype Build

Validates specification completeness, assembles spec files + stack patterns + agent rules into a single build prompt, tags the commit, and outputs a running prototype.

Learnings

Lorem ipsum dolor sit amet, the git tag as a build boundary was the key insight; without it there is no reliable way to distinguish what has been built from what is pending.

Inputs

  • spec_template/
  • Specifications/
  • Stack/
  • Technology_Rules/

Outputs

  • Prototype/
  • SCORECARD.md
  • git tag oneshot/{name}/{date}
Workflow 2

Change Iteration

Packages CHANGES.md with current prototype state into an iterate prompt; agent applies the change; squash-merges to project on completion.

flowchart LR classDef dir fill:#0a5c38,stroke:#2cb67d,color:#fff,font-weight:bold classDef md fill:#d4a017,stroke:#a07810,color:#111,font-weight:bold classDef script fill:#1e40af,stroke:#3b5fc0,color:#fff CH{{"CHANGES.md"}}:::md --> S1["iterate.sh"]:::script PT1(["Prototype/"]):::dir --> S1 SC1{{"SCORECARD.md"}}:::md --> S1 S1 --> PT2(["Prototype/"]):::dir S1 --> SC2{{"SCORECARD.md"}}:::md PT2 --> S2["merge.sh"]:::script SC2 --> S2 S2 --> PROJ(["Project"]):::dir
Learnings

Lorem ipsum dolor sit amet, the change-as-file pattern with presence-equals-pending was simpler than any ticket system and worked naturally with git history — rejected changes stay in the file with a rejection reason appended.

Inputs

  • CHANGES.md
  • Prototype/
  • SCORECARD.md

Outputs

  • Prototype/ (updated)
  • SCORECARD.md (updated)
  • Project (squash-merged)
Workflow 3

Rules Propagation

Generates CLAUDE_RULES.md from BUSINESS_RULES.md and distributes it to all projects via AGENTS.md injection; validates compliance per project.

flowchart LR classDef dir fill:#0a5c38,stroke:#2cb67d,color:#fff,font-weight:bold classDef md fill:#d4a017,stroke:#a07810,color:#111,font-weight:bold classDef script fill:#1e40af,stroke:#3b5fc0,color:#fff RULES(["Technology Rules"]):::dir --> S1["summarize_rules.sh"]:::script --> CR{{"CLAUDE_RULES.md"}}:::md --> S2["ProjectUpdate.sh"]:::script --> PROJ(["Project"]):::dir --> S3["ProjectValidate.sh"]:::script --> KPI{{"KPI.md"}}:::md
Learnings

Lorem ipsum dolor sit amet, keeping the rationale in BUSINESS_RULES.md and generating a short CLAUDE_RULES.md from it meant agents received a concise contract they actually followed, not a wall of text they pattern-matched around.

Inputs

  • BUSINESS_RULES.md
  • templates/
  • stack/

Outputs

  • CLAUDE_RULES.md
  • Project AGENTS.md (injected)
  • KPI.md
Workflow 4

Specification Review

AI-scored gap analysis of the specification directory — produces a prioritized gap list (P0–P4), quality scorecard, and next iteration prompt.

flowchart LR classDef dir fill:#0a5c38,stroke:#2cb67d,color:#fff,font-weight:bold classDef md fill:#d4a017,stroke:#a07810,color:#111,font-weight:bold classDef script fill:#1e40af,stroke:#3b5fc0,color:#fff SPEC(["Specifications/"]):::dir --> SI["spec_iterate.sh"]:::script --> GAPS{{"REFERENCE_GAPS.md"}}:::md SI --> SCORE{{"SPEC_SCORECARD.md"}}:::md SI --> ITER{{"SPEC_ITERATION.md"}}:::md
Learnings

Lorem ipsum dolor sit amet, asking the AI to review the spec before the build revealed blind spots a human reviewer would miss consistently — missing error states, unspecified edge cases, ambiguous field names.

Inputs

  • Specifications/
  • BUSINESS_RULES.md

Outputs

  • REFERENCE_GAPS.md
  • SPEC_SCORECARD.md
  • SPEC_ITERATION.md
Workflow 5

Session Capture

Reads Claude Code session logs from the prototype directory and writes the changes back as formal ticket files in the specification directory.

flowchart LR classDef dir fill:#0a5c38,stroke:#2cb67d,color:#fff,font-weight:bold classDef script fill:#1e40af,stroke:#3b5fc0,color:#fff PT(["Prototype/"]):::dir --> TL["tran_logger.sh"]:::script TL --> SPEC(["Specifications/"]):::dir
Learnings

Lorem ipsum dolor sit amet, formalizing session changes into tickets retroactively kept the spec honest — changes made outside the ticket workflow were no longer invisible or lost between sessions.

Inputs

  • Prototype/ (session logs)

Outputs

  • Specifications/ PATCH-NNN-tl-*.md
  • Specifications/ AC-NNN-tl-*.md
