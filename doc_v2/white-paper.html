Overview

The SDD Loop

Specification files are the single source of truth; prototype code is a derived artifact.

In Specification Driven Design (SDD), every change originates in a specification directory — structured markdown files that define screens, data models, features, and acceptance criteria. An AI agent reads these files and builds or updates the prototype. No ad-hoc instructions, no manual editing of generated code. The specification is what you author; the code is what the agent produces.

Fig 1 — The SDD loop: clockwise from SPECIFICATIONS through script steps to PROTOTYPE and back.

The outer ring shows the main objects: SPECIFICATIONS (what to build), STACK (how to build it), PROTOTYPE (the running result), and PROJECTS (the promoted, standards-conformant system). The amber cards between them are the script handshakes that move work from one state to the next. The PRINCIPLES in the center — simplifying, change management, strict lint, expose endpoints, data first — are the engineering constraints that every script and specification must satisfy.

Object

Specifications

Canonical markdown files that define the full intent of the prototype.

A specification directory contains one file per concern: screens, database schema, UI patterns, features, and acceptance criteria. Canonical files are permanent and authored by the human. Ticket files are numbered variants that represent a single pending change — their presence means pending; their deletion by the agent means done. Every canonical file ends with an ## Open Questions section, which blocks tickets for that file from being applied until the question is resolved. This forces specification precision before execution.

File Types

  • SCREEN-{Name}.md
  • DATABASE.md
  • UI.md
  • ARCHITECTURE.md
  • FEATURE-{Name}.md
  • METADATA.md

Ticket Types

  • SCREEN-NNN-*.md
  • FEATURE-NNN-*.md
  • PATCH-NNN-*.md
  • AC-NNN-*.md
  • INTENT-NNN-*.md
Learnings

Lorem ipsum dolor sit amet, the mandatory Open Questions section felt like friction at first; it eliminated an entire class of ambiguous builds that would have wasted agent time.

Object

Stack

Prescriptive technology patterns that cap context and eliminate tech-decision churn.

The stack directory holds one file per technology — Flask, SQLite, Bootstrap5, security patterns, pipeline templates. These files are included verbatim in every build prompt. Their purpose is dual: they give the AI agent copy-paste implementation patterns, and they cap the token budget by making the answer to "how do I do X?" a file read rather than a freeform generation. Context minimization is an engineering discipline, not an afterthought.

Reads

  • RulesEngine/stack/flask.md
  • RulesEngine/stack/sqlite.md
  • RulesEngine/stack/styles.md
  • RulesEngine/stack/security.md
  • RulesEngine/stack/pipelines.md

Effect

  • Agent uses prescribed patterns
  • No tech-choice variance per build
  • Prompt token budget bounded
  • Stack decisions made once, reused everywhere
Learnings

Lorem ipsum dolor sit amet, writing the stack files once and reusing them across prototypes cut per-build prompt size by roughly a third and removed inconsistent library choices.

Script

setup.sh

Scaffold a new specification directory from templates — one command, complete structure.

Reads

  • ProjectName argument

Writes

  • Specifications/{Name}/ directory
  • METADATA.md (stubbed)
  • README.md, ARCHITECTURE.md
  • ACCEPTANCE_CRITERIA.md
  • IDEAS.md, DEPLOY_LOG.md
Learnings

Lorem ipsum dolor sit amet, scaffolding enforced structure from the first commit; projects started with setup.sh had consistently better validate.sh pass rates on first run.

Script — Quality Gate

validate.sh

Verifies specification completeness before any build prompt is generated.

Quality in SDD is defined at the specification level, not the code level. validate.sh checks required files, METADATA fields, Open Questions sections, stack file existence, naming conventions, and template cleanup. Exit 0 means the specification is ready to build. Exit 1 means the human has more work to do. No agent is involved — this is a deterministic lint pass over structured files.

Reads

  • Specifications/{Name}/ all files
  • METADATA.md fields
  • RulesEngine/stack/ (existence check)

Outputs

  • Exit 0 — specification valid
  • Exit 1 — errors to stderr
  • Conformity level score
Learnings

Lorem ipsum dolor sit amet, gating on specification quality rather than code quality surfaced gaps earlier and made the AI's output predictably better on the first attempt.

Script

oneshot.sh

The canonical build command — validate, tag, and generate the complete build prompt in one step.

The oneshot build assembles CLAUDE_RULES, ONESHOT_BUILD_RULES, all stack files, and every specification file into a single prompt. The AI agent receives this prompt and builds the prototype from scratch. The specification commit is tagged (oneshot/{project}/{date.N}) so any future iterate run can diff against the exact state used for the build. Tags are permanent git objects — never pruned.

Reads

  • RulesEngine/CLAUDE_RULES.md
  • RulesEngine/ONESHOT_BUILD_RULES.md
  • RulesEngine/stack/ (all)
  • Specifications/{Name}/ (all)

Writes

  • {Name}/oneshot-prompt.md
  • git tag oneshot/{Name}/{date}
  • {Name}/.env (build metadata)
  • {Name}/DEPLOY_LOG.md (appended)
Learnings

Lorem ipsum dolor sit amet, the git tag as a build boundary was the key insight; without it there is no reliable way to distinguish what has been built from what is pending.

Script — AI-Assisted

spec_iterate.sh

Ask the AI what the specification is missing before writing more tickets.

Before each iteration cycle, spec_iterate.sh feeds all specification files and BUSINESS_RULES into an AI agent and asks it to rate the spec quality across seven dimensions, identify unspecified gaps by priority (P0–P4), and draft the next one or two spec files to write. The human reviews and edits the output before proceeding. This closes the loop between the AI's view of the spec and the human's intent — without requiring the human to anticipate every gap themselves.

Reads

  • Specifications/{Name}/ (all)
  • RulesEngine/BUSINESS_RULES.md

Writes

  • REFERENCE_GAPS.md (gap list, P0–P4)
  • SPEC_SCORECARD.md (7-dimension rating)
  • SPEC_ITERATION.md (next prompt)
Learnings

Lorem ipsum dolor sit amet, letting the AI score its own input specification revealed blind spots that a human reviewer would consistently miss, particularly around error states and edge conditions.

Script

iterate.sh + tran_logger.sh

Incremental change management — each change expressed as a numbered ticket, applied by the agent, tracked by the specification.

After the initial build, every change enters the system as a ticket file. The ticket number (NNN) determines application order. iterate.sh reads all ticket files added since the last oneshot tag, assembles them with ARCHITECTURE and ACCEPTANCE_CRITERIA, and generates the iterate prompt. The agent applies each ticket, validates it, and on success deletes the ticket file. Rejected tickets get a ## Rejection Reason section added and remain for the author to resolve.

tran_logger.sh reads Claude Code session logs and converts interactive changes into ticket files automatically — closing the loop between informal exploration and the formal specification history.

Reads

  • Ticket files since last oneshot tag
  • ARCHITECTURE.md
  • ACCEPTANCE_CRITERIA.md
  • Claude session logs (tran_logger)

Writes

  • {Name}/iterate-prompt.md
  • PATCH-NNN-tl-*.md (tran_logger)
  • AC-NNN-tl-*.md (tran_logger)
  • Prototype docs/SCORECARD.md
Learnings

Lorem ipsum dolor sit amet, the ticket-as-file pattern with presence-equals-pending was simpler than any database-backed ticket system and worked naturally with git history and code review tooling.

Object

Rules Engine

A single source of behavioral contracts, distributed automatically to every project.

The Rules Engine is a directory of files in the Specifications repository. BUSINESS_RULES.md is the authoritative source — edited by the human, containing full rationale. CLAUDE_RULES.md is generated from it by summarize_rules.sh and is injected into every project's AGENTS.md. Projects never edit their CLAUDE_RULES block; they receive it from the distribution pipeline. This means behavioral standards are set once and applied uniformly — without manual compliance work per project.

Reads

  • RulesEngine/BUSINESS_RULES.md
  • RulesEngine/ONESHOT_BUILD_RULES.md
  • RulesEngine/templates/ (3 files)
  • RulesEngine/stack/ (14 files)

Writes / Distributes

  • RulesEngine/CLAUDE_RULES.md (generated)
  • Each project's AGENTS.md (injected)
  • Each project's bin/common.sh
  • Each project's bin/common.py
Learnings

Lorem ipsum dolor sit amet, separating the rationale document from the injected contract kept CLAUDE_RULES.md concise enough that agents actually followed it, rather than pattern-matching on length.

Living Example

GAME — Framework Managing Itself

The dashboard that discovers, validates, and operates all other projects was itself built using SDD.

GAME (GEM Agentic Management Engine) is a Flask/SQLite dashboard that reads project METADATA.md files, validates their CLAUDE_RULES conformance levels, exposes their health endpoints, and surfaces their documentation. It was built from specification files in this repository using the same oneshot/iterate workflow it exists to manage. When new features were identified through spec_iterate.sh gap analysis, they were added as ranked ticket files, applied by the agent, and promoted through the merge workflow. No manual code editing. The specification led; the code followed.

Learnings

Lorem ipsum dolor sit amet, building the framework tool with the framework process provided continuous real-world validation; gaps in the methodology surfaced immediately as specification gaps in GAME itself.
