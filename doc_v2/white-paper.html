Workflow 1

Prototype Build

Scaffold a specification directory, validate it, then generate a single build prompt the agent uses to produce a running prototype from scratch.

Learnings

Lorem ipsum dolor sit amet, the git tag as a build boundary was the key insight; without it there is no reliable way to distinguish what has been built from what is pending.

Inputs

  • spec_template/ (scaffolding)
  • Specifications/ (authored files)
  • Stack/ (technology patterns)
  • Technology_Rules/ (CLAUDE_RULES.md)

Outputs

  • Prototype/ (running codebase)
  • SCORECARD.md
  • git tag oneshot/{name}/{date}
Workflow 2

Change Iteration

Express each change as a file. The agent applies it against the prototype, scores the result, and merges to the project branch when ready.

flowchart LR classDef dir fill:#0a5c38,stroke:#2cb67d,color:#fff,font-weight:bold classDef md fill:#d4a017,stroke:#a07810,color:#111,font-weight:bold classDef script fill:#1e40af,stroke:#3b5fc0,color:#fff CH{{"CHANGES.md"}}:::md --> S1["iterate.sh"]:::script PT1(["Prototype/"]):::dir --> S1 SC1{{"SCORECARD.md"}}:::md --> S1 S1 --> PT2(["Prototype/"]):::dir S1 --> SC2{{"SCORECARD.md"}}:::md PT2 --> S2["merge.sh"]:::script SC2 --> S2 S2 --> PROJ(["Project"]):::dir
Learnings

Lorem ipsum dolor sit amet, the ticket-as-file pattern with presence-equals-pending was simpler than any database-backed ticket system and worked naturally with git history and code review tooling.

Inputs

  • CHANGES.md (ticket)
  • Prototype/ (current state)
  • SCORECARD.md

Outputs

  • Prototype/ (updated)
  • SCORECARD.md (updated)
  • Project (squash-merged branch)
Workflow 3

Rules Propagation

Author behavioral contracts once; distribute them automatically to every project and verify compliance.

flowchart LR classDef dir fill:#0a5c38,stroke:#2cb67d,color:#fff,font-weight:bold classDef md fill:#d4a017,stroke:#a07810,color:#111,font-weight:bold classDef script fill:#1e40af,stroke:#3b5fc0,color:#fff RULES(["Technology Rules"]):::dir --> S1["summarize_rules.sh"]:::script --> CR{{"CLAUDE_RULES.md"}}:::md --> S2["ProjectUpdate.sh"]:::script --> PROJ(["Project"]):::dir --> S3["ProjectValidate.sh"]:::script --> KPI{{"KPI.md"}}:::md
Learnings

Lorem ipsum dolor sit amet, separating the rationale document from the injected contract kept CLAUDE_RULES.md concise enough that agents actually followed it, rather than pattern-matching on length.

Inputs

  • BUSINESS_RULES.md (source of truth)
  • templates/ (common.sh, common.py)
  • stack/ (technology patterns)

Outputs

  • CLAUDE_RULES.md (generated)
  • Project AGENTS.md (injected)
  • KPI.md (compliance report)
Workflow 4

Specification Review

Run the AI against the specification itself — before writing tickets — to surface gaps, score quality, and draft the next specification files.

flowchart LR classDef dir fill:#0a5c38,stroke:#2cb67d,color:#fff,font-weight:bold classDef md fill:#d4a017,stroke:#a07810,color:#111,font-weight:bold classDef script fill:#1e40af,stroke:#3b5fc0,color:#fff SPEC(["Specifications/"]):::dir --> SI["spec_iterate.sh"]:::script --> GAPS{{"REFERENCE_GAPS.md"}}:::md SI --> SCORE{{"SPEC_SCORECARD.md"}}:::md SI --> ITER{{"SPEC_ITERATION.md"}}:::md
Learnings

Lorem ipsum dolor sit amet, letting the AI score its own input specification revealed blind spots that a human reviewer would consistently miss, particularly around error states and edge conditions.

Inputs

  • Specifications/ (all files)
  • BUSINESS_RULES.md

Outputs

  • REFERENCE_GAPS.md (P0–P4 gaps)
  • SPEC_SCORECARD.md (7-dimension rating)
  • SPEC_ITERATION.md (next prompt)
Workflow 5

Session Capture

Convert informal Claude Code session activity into formal specification ticket files automatically, closing the loop between exploration and the spec history.
