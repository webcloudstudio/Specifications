Engineering Methodology

Specification Driven Design

A repeatable process for building software with AI agents — where the spec files are the product and the code is what the agent produces.
Ed Barlow Web Cloud Studio 2026

Specification Driven Design (SDD) is a development process built around one rule: you never edit generated code directly. Every change — a new screen, a bug fix, a data model change — starts as a markdown file in a specification directory. An AI agent reads those files and builds or updates the prototype. The spec is what you maintain; the code is what the agent produces from it.

This matters because AI agents are good at executing clear instructions and bad at holding context across sessions. SDD gives the agent everything it needs in a single, structured prompt — and gives you a process that works the same way every time, whether you're starting a new project or shipping a change to an existing one.

Workflow 1

Prototype Build

You write markdown files describing what to build — screens, data models, API routes, acceptance criteria. setup.sh creates the directory structure from a template. validate.sh checks every required field and file exists before you waste an agent run on an incomplete spec. oneshot.sh then assembles your spec files, the technology stack patterns, and the agent behavior rules into a single prompt. You paste that into Claude Code and the agent builds a working prototype.

Workflow 2

Change Iteration

Once the prototype exists, changes don't go directly to the code. You write a CHANGES.md file describing what needs to change and why. iterate.sh packages that file with the current prototype state and scorecard into a prompt, and the agent applies the change. When it's validated, merge.sh squash-merges the feature branch into the project — one commit per change, clean history. The old prototype state and scorecard become inputs to the next iteration.

Lorem ipsum dolor sit amet, the change-as-file pattern with presence-equals-pending was simpler than any ticket system and worked naturally with git history — rejected changes stay in the file with a rejection reason appended.

  • SCORECARD.md

  • SCORECARD.md (updated)
  • Project (squash-merged branch)
Workflow 3

Rules Propagation

Every project built with SDD shares the same agent behavior rules — how to structure code, what conventions to follow, what to never do. Those rules live in one file (BUSINESS_RULES.md) in the Specifications repository. summarize_rules.sh generates a concise version that gets injected into every project's AGENTS.md automatically. ProjectUpdate.sh pushes the update out. ProjectValidate.sh checks each project is compliant and writes a KPI report. You change the rules once; every project gets them.

Learnings

Lorem ipsum dolor sit amet, keeping the rationale in BUSINESS_RULES.md and generating the short CLAUDE_RULES.md from it meant agents received a concise contract they actually followed, not a wall of text they pattern-matched around.

Inputs

  • BUSINESS_RULES.md
  • templates/ (common.sh, common.py)
  • stack/ (technology patterns)

Outputs

  • CLAUDE_RULES.md (generated)
  • Project AGENTS.md (injected)
  • KPI.md (compliance report)
Workflow 4

Specification Review

Before writing more change files, you run spec_iterate.sh. It feeds your entire spec directory to an AI that reads it the same way the build agent will — and tells you what's missing. The output is a scored gap report (P0 critical to P4 nice-to-have), a quality scorecard across seven dimensions, and a draft of the next spec files you should write. This catches gaps before the agent builds them wrong, not after.

Learnings

Lorem ipsum dolor sit amet, asking the AI to review the spec before the build revealed blind spots a human reviewer would miss consistently — missing error states, unspecified edge cases, ambiguous field names.

Inputs

  • Specifications/ (all files)
  • BUSINESS_RULES.md

Outputs

  • REFERENCE_GAPS.md (P0–P4 gaps)
  • SPEC_SCORECARD.md (quality rating)
  • SPEC_ITERATION.md (next prompt)
Workflow 5

Session Capture

Developers make informal changes — test something in a session, fix a bug interactively, try an idea. Without a process, those changes exist in the code but not in the spec. tran_logger.sh reads Claude Code session logs and converts the changes made during a session into formal ticket files, written back into the spec directory. Nothing that happened in a session goes unrecorded.

Lorem ipsum dolor sit amet, formalizing session changes into tickets retroactively kept the spec honest — changes made outside the ticket workflow were no longer invisible or lost between sessions.
